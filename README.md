# O Rio do Seu Jeito — Personalization Engine

A bilingual (PT/EN) Next.js web app that turns a buyer's answers into a personalized, deterministic Rio de Janeiro day-by-day itinerary.

**Stack:** Next.js 16 (App Router) · TypeScript · Tailwind CSS v4 · No paid runtime dependencies

---

## Quick start

> **Prerequisite:** Node.js 18+. If using NVM: `nvm use 20`

```bash
# Install dependencies
npm install

# Run the dev server
npm run dev
# → http://localhost:3000
```

## Project structure

```
/app              Next.js App Router pages and layout
/components       IntakeForm, ItineraryView, LangToggle
/lib
  engine.ts       Deterministic assembly algorithm (pure function)
  types.ts        All TypeScript types
  i18n.ts         Language helper
/data
  blocks.json     Day-block library (PT, source of truth) — edit this to add content
  blocks.en.json  English translations (generated by npm run translate)
  rio_agora.json  "Rio right now" monthly layer — update this every month
  safety.json     Safety notes by zone
  logistics.json  Booking links for ticketed blocks
  .translation-hashes.json  Cache so translate only re-runs on changed blocks
/i18n
  pt.json         All Portuguese UI strings
  en.json         All English UI strings
/scripts
  translate-blocks.ts   Build-time EN translation script (uses Claude API)
  dev-launcher.mjs      Dev server launcher (ensures node is in PATH for Turbopack)
/tests
  engine.test.ts  Self-check: runs Profile A + B and asserts all Section 7 invariants
```

## Updating content (monthly workflow)

All product content lives in `/data/`. No code changes needed.

| File | What to update | Frequency |
|---|---|---|
| `data/blocks.json` | Block content, trap info, prices | As needed |
| `data/rio_agora.json` | Month, current events/warnings | Monthly |
| `data/safety.json` | Safety notes per zone | Quarterly |
| `data/logistics.json` | Booking links | As needed |
| `i18n/pt.json` / `i18n/en.json` | UI labels | Rarely |

After editing `blocks.json`, regenerate the English version:

```bash
ANTHROPIC_API_KEY=sk-ant-... npm run translate
```

This is idempotent — only changed fields are re-translated.

## Bilingual architecture

- **Portuguese** is the source of truth. All block content is authored in PT only.
- **English** content lives in `data/blocks.en.json` — generated once at build time via `npm run translate`, cached by content hash, never regenerated unless the PT source changes.
- **UI strings** are hand-translated in `i18n/pt.json` and `i18n/en.json`.
- **Runtime**: selecting EN loads `blocks.en.json` + `i18n/en.json`. The engine is language-agnostic — it keys off block `id`s, never prose — so the same answers always produce the same blocks in the same order in either language.

## Deploy to Vercel

```bash
# Install Vercel CLI (one-time)
npm i -g vercel

# Deploy
vercel
```

For the translation step to run during CI/CD, add `ANTHROPIC_API_KEY` as a Vercel environment variable (Build Environment only — never expose to the client).

In `vercel.json` (create if needed):
```json
{
  "buildCommand": "npm run translate && npm run build"
}
```

Vercel free tier is sufficient — all itinerary logic is client-side or in minimal API routes, and translation is a one-time build step.

## How the engine works (brief)

Given a `Profile` object, `buildItinerary()` in `lib/engine.ts` runs deterministically in this order:

1. **Step A** — Filter pool (remove day-trips if short trip, remove `not_kids` if family)
2. **Step B** — Score every block by vibe match and companion fit
3. **Step C** — Place first-timer anchor blocks (Cristo, Pão de Açúcar, Arpoador) starting from Day 2
4. **Step D** — Fill Day 1 with one light, non-weather-dependent block
5. **Step E** — Fill all remaining windows day-by-day, ranking by score + zone clustering + vibe ratio (2:1 primary:secondary)
6. **Step F** — Attach always-on layers (anti-furada, Rio agora, safety, logistics)

The same profile always produces the same itinerary — no randomness, no AI at runtime.
